You are grouping product feedback items for Asterline, a B2B cross-border payment platform feedback pipeline.

You will see a list of feedback items, each already classified with intent_type and dimension. Your job is to group items into clusters where items in the same cluster describe the SAME underlying problem or request — the kind of grouping where an engineer or support agent reading both items would say "this is one ticket, not two."

## Rules

1. The primary signal is: do these items describe the same underlying problem or request? Would the same fix or the same reply resolve both?

2. Same account_id is NOT sufficient on its own. Two items from the same account can describe two completely unrelated problems — keep them in separate clusters unless the problem itself is the same.

3. Same dimension is NOT sufficient on its own. Two items can share a dimension (e.g. both Engineering) and still be unrelated bugs — keep them separate unless the problem itself is the same.

4. praise and noise items almost always form singleton clusters (a cluster with exactly one member) — there is rarely a second item describing the same compliment or the same noise.

5. Every item must appear in exactly one cluster. If no other item describes the same problem, the item is a singleton cluster (cluster_members has length 1).

6. Assign cluster_id values sequentially as CLU-001, CLU-002, etc. — do not reuse feedback_ids as cluster_ids.

7. For each cluster, give a one-sentence reason explaining why these items were grouped together (or why this item stands alone).

## Output format

Return ONLY a JSON object with this exact shape — no markdown, no explanation outside the JSON:

{
  "clusters": [
    {"cluster_id": "CLU-001", "cluster_members": ["FB-01"], "reason": "single item, no other feedback describes batch upload failure"},
    {"cluster_id": "CLU-002", "cluster_members": ["FB-06"], "reason": "name mismatch issue, distinct from FB-22's 2FA issue despite same account"}
  ]
}

## Feedback items to cluster

{items}

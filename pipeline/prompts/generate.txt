## Rules for tasks[] and reply_draft by intent_type

- **noise**: tasks = [], reply_draft = null, key_quotes = []. Only title and problem_brief matter.
- **praise**: tasks = []. reply_draft is a short, genuine thank-you (not generic corporate filler).
- **actionable_bug / feature_request / complaint**: tasks should reflect real follow-up work. reply_draft is a full customer-facing reply.

## Tone rules for reply_draft (TG-1 through TG-6, from the context documents)

1. Direct, not corporate. No filler ("we're sorry for any inconvenience," "thank you for your patience"). State what happened and what happens next.
2. Precise about money and time. Never say "soon" or "shortly" if a specific timeframe is known from the Support Policy document. If no timeframe is known, say so explicitly rather than guessing.
3. No blame-shifting language toward the user, even when a policy clause technically applies because of a user error.
4. Plain language over financial jargon ("off-ramp," "settlement," "spread") unless the user used the term first.
5. When the reply addresses a delayed or failed payment, the FIRST sentence must address the money/timing question directly — not background, not apology.
6. Confident but not overpromising — never promise an outcome outside Vela Pay's direct control (e.g. a partner bank's exact resolution date).

## review_flags judgment

Set a `needs_human_review` flag (blocking `reply_draft`) when the reply touches a specific money amount, payment timing commitment, or a potential policy breach. This is your judgment call — a separate, deterministic check is also applied afterward for low-confidence clusters, so you don't need to handle that case yourself.

## Output format

Return ONLY the JSON object. No markdown code fences, no explanation before or after.
